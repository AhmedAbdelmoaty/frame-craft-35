# Ready State

## Am I Ready?

Yes. I am ready to act as a game-development assistant for a non-technical user, with the current limitation that the actual game concept has not been designed yet.

## What I Now Know

I know how to:

* choose skills automatically from the task type
* translate weak/non-technical requests into practical workflows
* avoid copying reference-project style
* use reference projects as workflow material only
* plan before complex implementation
* generate assets through anchor-based workflows
* distinguish image-generation tasks from video-generation animation tasks
* recover frames from generated pose boards
* curate animation frames
* normalize spritesheets
* check frame dimensions, drift, bleeding, transparency, and alignment
* use browser/playtest workflows when visual behavior matters
* avoid duplicate skill creation unless there is clear value

## What Is Still Missing

The project still needs:

* game vision
* target learner
* statistics/business topic
* core loop
* success and failure rules
* art direction
* asset index structure for the new game
* prototype plan

## What I Will Do Automatically On Future Tasks

For every task, I will:

1. classify the request
2. choose the relevant skill/workflow
3. inspect local context
4. plan when needed
5. avoid touching unrelated files
6. use asset and spritesheet checklists for generated assets
7. use visual testing when needed
8. document prompts/plans/decisions when useful
9. avoid copying reference content as final style

## First Step After This Phase

The next step should be a Game Vision Workshop, but only when the user is ready.

Before that, no gameplay should be built and no final art direction should be assumed.
