# Analyst Triage — No-Go Rules & Acceptance Criteria

## 1. No-Go Rules

### No Quiz

Forbidden:

- multiple-choice questions as main gameplay;
- “Which team is better?” direct question;
- correct/wrong screen.

Correct alternative:

- player assigns recommendations and sees consequences.

### No Dashboard Core

Forbidden:

- dense KPI panels;
- long tables;
- chart-only gameplay;
- BI dashboard clone.

Correct alternative:

- employees as units;
- spatial threshold;
- animated distribution;
- tactile analysis tools.

### No Case Study Worksheet

Forbidden:

- read scenario then choose answer;
- report page as main action;
- static file review.

Correct alternative:

- pressure, movement, scan, sort, mark, test, recommend.

### No Walking Simulator

Forbidden:

- rooms exist only to collect files;
- long walking before actual gameplay;
- NPCs explain the solution.

Correct alternative:

- rooms provide pressure, constraints, or tool unlocks quickly.

### No Magic Tool

Forbidden:

- tool says “Team B is correct”;
- tool labels “misleading mean” before player interprets;
- tool auto-solves recommendation.

Correct alternative:

- tools reveal shape and sensitivity only.

### No Overbuilding

Forbidden:

- full map before core loop;
- final AI art before V1;
- many missions before V1 fun;
- economic systems before feedback loop.

Correct alternative:

- build minimum playable moment first.

## 2. Prototype V1 Acceptance Criteria

V1 is acceptable only if:

- player starts inside a company context;
- there is a visible meeting pressure/timer;
- Sales and HR create different pressures;
- player reaches Analyst Office;
- two teams are visible as units, not a table;
- player can Scan;
- player can Sort;
- player can place or trigger Threshold;
- player can mark outliers;
- player can run Impact Test;
- player can assign Reward and Training;
- correct and wrong decisions produce different world feedback;
- the game never says “correct/wrong” as primary feedback.

## 3. Fun Acceptance Criteria

The prototype is fun enough to continue if:

- player feels pressure;
- player touches/manipulates things;
- player sees data transform;
- player experiences an “aha” moment;
- player sees consequences;
- replaying once feels useful.

## 4. Learning Acceptance Criteria

The prototype teaches correctly if:

- player understands that mean can mislead;
- player does not learn that mean is always bad;
- player sees threshold and distribution as decision-critical;
- player makes a better recommendation using statistics;
- feedback comes from consequences, not lecture.

## 5. Failure Signals

Stop and redesign if:

- the screen looks like a dashboard;
- most time is spent reading;
- player has no action for more than 20 seconds;
- NPC text explains the solution;
- decision is just choosing A or B;
- wrong outcome is only a text message;
- Codex adds complex systems before V1 works.
